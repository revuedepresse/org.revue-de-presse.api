# Symfony 7.4 LTS + Doctrine ORM 3 migration — design

**Status:** draft for review
**Date:** 2026-05-06
**Project:** `revuedepresse/http-api` (this repository)
**Author:** brainstorming session, this branch

---

## 1. Goal and non-goals

### Goal

Move the application from **Symfony 5.4 LTS** to **Symfony 7.4 LTS**, with **Doctrine ORM 2 → 3** as a sequenced sub-migration. The motivation is feature-pull (modern framework features, attribute-based mapping, current ecosystem) rather than EOL pressure.

### Non-goals

- Symfony 8.x. Symfony 8.0 (released Nov 2025) is on the short-support track (PHP 8.4+ floor, 8-month support windows). Re-evaluate once Symfony 8.4 LTS lands (~Nov 2027).
- Test-coverage expansion beyond what each stage strictly needs to validate its own work.
- Refactoring unrelated to the migration. Targeted improvements within files we touch are fair game; broad cleanup is not.
- Replacing other libraries (`abraham/twitteroauth`, `predis/predis`, etc.) unless the migration forces it.

---

## 2. Current state — evidence

- **Symfony:** all `symfony/*` components on `^5.4` (Flex layout).
- **PHP constraint:** `composer.json` `require.php: ">=8.1"`, `config.platform.php: "8.1.6"`. Local dev runtime is PHP 8.5.5; the active container runtime appears to be PHP 8.4 (per the xdebug `zend_extension` path `no-debug-non-zts-20240924` in `provisioning/containers/service/templates/extensions.ini.dist`).
- **Doctrine:** `doctrine/orm: ^2.8`, `doctrine/doctrine-bundle: ^2.2`, `doctrine/annotations: ^2.0`. Mapping is split: 4 entity groups use `type: annotation` (9 PHP files: `Token`, `Whisperer`, `ArchivedStatus`, `TokenType`, `Status`, `Publication`, `MembersList`, `Role`, `Member`); 2 entity groups use `type: xml` (under `config/model/curation/` and `config/model/trends/`). Zero `#[ORM\…]` attributes today.
- **Routing:** zero `@Route` annotations or `#[Route]` attributes in `src/`. All routing is YAML in `config/routes/`.
- **Test coverage:** thin but representative — auth/ingestion/API endpoints are minimally covered. Full suite passes 12/12 as of this branch.
- **Doctrine migrations:** `doctrine/doctrine-migrations-bundle` is wired but **zero migration files exist**. Schema is presumably created via `doctrine:schema:update` or external SQL.
- **Doctrine extension packages:** both `oro/doctrine-extensions` and `beberlei/doctrineextensions` are installed and registered in `config/packages/doctrine.yaml`'s `doctrine_extensions_dql` map. Initial grep finds **zero use of either namespace** in `src/` and **zero DQL invocations** of their function names in `src/` (only `Scienta\DoctrineJsonFunctions\…` JSON_EXTRACT is actually called — by `PublicationRepository.php`). XML mappings still need confirmation (see Stage 2). Both packages are candidates for removal.

## 3. Target state

- **Symfony 7.4 LTS** with all `symfony/*` on `^7.4`.
- **PHP** runtime ≥ **8.2** (Symfony 7 floor); composer platform pin moved to a concrete `8.2.x`.
- **Doctrine ORM 3**, attribute-based PHP mapping, XML mappings retained for the two existing XML-mapped groups.
- **`doctrine/annotations`** removed entirely.
- Both `oro/doctrine-extensions` and `beberlei/doctrineextensions` removed (assuming Stage 2's deeper verification confirms the XML mappings don't reference them); related dep ecosystem bumped to ORM-3-compatible majors.
- Two new dev tools in place — **Rector** (annotations → attributes; Symfony version-bridge sets) and **PHPStan** (level 5, with `phpstan-baseline.neon` capturing pre-existing issues).
- Unchanged: framework-level architecture (controllers, routing layout, repository pattern, security model, Twig presence, Messenger).

---

## 4. Approach — selected and rationale

Three sequencing approaches were considered:

- **A) Hop-by-hop with Doctrine in the middle:** 5.4 → 6.4 → ORM 3 → 7.4. **Selected.** ORM 3 lands on a stable Symfony 6.4 base; old ORM is carried across only one Symfony hop; the final Symfony 7.4 hop is framework-only.
- B) Doctrine first on 5.4, then Symfony hops. Removes annotations earliest but layers ORM 3 onto a framework that already has PHP 8.4-induced deprecation noise.
- C) Symfony hops first, ORM 3 last. Maximum axis isolation but carries `doctrine/orm 2` and the annotation footprint across two Symfony majors.

### Stage 1 — Symfony 5.4 → 6.4 LTS

**PHP runtime stays at the existing version (≥ 8.1). composer.json `php` constraint stays `>=8.1`.**

- **Composer changes.** All `symfony/*: ^5.4` → `^6.4`. `symfony/translation-contracts: ^2.3` → `^3`. Twig already permits 3.x, lock will resolve naturally.
- **Config changes — biggest friction is `security.yaml`.** Symfony 6 removes the deprecated guard-authenticator system in favor of the new authenticator-based system (which has been available since 5.3 as opt-in). The current firewall uses `guard.authenticators: [app.token_authenticator]`; Stage 1 must port `app.token_authenticator` to implement `Symfony\Component\Security\Http\Authenticator\AuthenticatorInterface`. **This is the highest-risk change in the stage** because production auth has thin test coverage; the local controller tests (already added) bypass auth entirely via `firewalls.main.security: false` in test env, so they don't catch authenticator regressions.
- **Other config touch points likely:** `framework.yaml` (default values realignment), bundle registrations, kernel attribute usage. `bin/console debug:container --deprecations` is the steering instrument.
- **Tooling — Rector with `rector/rector-symfony`.** Run `SymfonySetList::SYMFONY_60`, `SYMFONY_61`, `SYMFONY_62`, `SYMFONY_63`, `SYMFONY_64` sequentially. Each set as its own commit; review every diff before applying.
- **Manual review areas.** Event subscribers using string event names → class FQCNs; Messenger handler interfaces; CORS trait if it touches symfony/http-foundation internals.

### Stage 2 — Doctrine ORM 2 → 3

**Symfony stays at 6.4. PHP runtime unchanged.**

- **Composer changes.** `doctrine/orm: ^2.8` → `^3`; `doctrine/doctrine-bundle: ^2.2` → `^2.10` minimum (already current major; ORM 3 supported from 2.10+); `ramsey/uuid-doctrine: ^1.8` → `^2`; `scienta/doctrine-json-functions: ^4.1` → `^6`. Drop `doctrine/annotations` entirely. Drop `oro/doctrine-extensions` and `beberlei/doctrineextensions` **after** verifying — see below.
- **Pre-stage verification of `oro` / `beberlei` removability.** This design records initial findings (zero PHP namespace usage; zero DQL invocations of their function catalog in `src/` PHP files; only `Scienta\…\JsonExtract` is actually used). Stage 2 must additionally check `config/model/curation/*.xml` and `config/model/trends/*.xml` for any `<named-query>` or function references using the catalog (e.g., `MATCH_AGAINST`, `CONVERT_TZ`, `DATE_FORMAT` as DQL functions). If the XML mappings are clean, both packages drop. If any XML references a function from the catalog, that function's package stays (or the XML mapping is rewritten).
- **Code changes — annotations to attributes for 9 entities.** Files: `Token`, `Whisperer`, `ArchivedStatus`, `TokenType`, `Status` (in `src/Twitter/Infrastructure/Http/Entity/`); `Publication` (in `src/Twitter/Domain/Publication/Entity/`); `MembersList` (in `src/Ownership/Domain/Entity/`); `Role`, `Member` (in `src/Membership/Domain/Entity/Legacy/`). XML-mapped entities (Curation, Trends groups) are unchanged.
- **Tooling — Rector with `rector/rector-doctrine`.** `DoctrineSetList::ANNOTATIONS_TO_ATTRIBUTES`. Reviewable diffs, idempotent.
- **Config change.** Flip the four `type: annotation` blocks in `config/packages/doctrine.yaml` to `type: attribute`. XML blocks unchanged. `doctrine_extensions_dql` parameter trimmed (or removed) based on the verification outcome above.
- **Removed/renamed APIs to grep and fix.** `EntityManager::merge` (removed), `Doctrine\ORM\Mapping\ClassMetadataInfo` (now `ClassMetadata`), `Doctrine\Common\Persistence\*` (now `Doctrine\Persistence\*`), lifecycle event listeners using string event names. Codebase grep at the start of the stage; fixes inline.
- **XML schema URL.** ORM 3 ships an updated XSD; `config/model/*.xml` files may need their `xsi:schemaLocation` adjusted. Quick fix when surfaced.

### Stage 3 — Symfony 6.4 → 7.4 LTS

**Runtime PHP floor moves to 8.2.**

- **Provisioning change first, in its own commit, ahead of the framework bump within the stage.**
  - `composer.json`: `config.platform.php` from `8.1.6` → `8.2.x` (a concrete PHP 8.2 patch). `require.php` from `>=8.1` → `>=8.2`.
  - `provisioning/containers/service/templates/extensions.ini.dist` audit:
    - The `zend_extension` path for xdebug (currently `/usr/local/lib/php/extensions/no-debug-non-zts-20240924/xdebug.so`, a PHP 8.4 API hash) must match whatever the active container PHP version is. If the container was already on PHP ≥ 8.2, this file may need no functional change; just verify the path matches the runtime. If a base image bump is required for production parity, update accordingly.
    - The `error_reporting = 'E_ALL & ~E_DEPRECATED & ~E_USER_DEPRECATED'` directive remains correct (production-style).
    - DataDog, timezone, PHP behavior settings stay as-is.
  - If a separate Dockerfile or container definition pins PHP elsewhere in `provisioning/`, bump it too.
- **Composer framework bump.** All `symfony/*: ^6.4` → `^7.4`. Symfony 7.0 = 6.4 minus deprecations, so this stage is mostly mechanical if Stage 1 left zero deprecations.
- **Tooling — Rector.** `SymfonySetList::SYMFONY_70`, `SYMFONY_71`, `SYMFONY_72`, `SYMFONY_73`, `SYMFONY_74` sequentially.
- **Polyfill cleanup.** `symfony/polyfill-php80`, `polyfill-php81` are no longer required at PHP ≥ 8.2; composer's resolver will leave them in `vendor/` only if a transitive dep still asks for them. After the framework bump, run `composer why symfony/polyfill-php80` etc. and remove transitively-orphaned entries by re-resolving (`composer update --with-all-dependencies symfony/*`).

## 6. Verification gates

Each stage exit requires **all** of the following.

### Common to every stage exit

- `phpunit` green; pass count ≥ Stage 0 baseline (12 at time of writing).
- `vendor/bin/phpstan analyse` (or `bin/phpstan` per the project's `bin-dir: bin` composer convention) clean — no new issues vs. `phpstan-baseline.neon`.
- `composer audit` reports no security advisories.
- `bin/console cache:clear` clean across `dev`, `test`, `prod` envs.
- `bin/console debug:container --deprecations` reviewed; new deprecations either fixed or explicitly deferred with a written reason in the PR description.

### Stage 0 specific

- `phpstan-baseline.neon` generated and committed; subsequent stages must not add to it.
- Deprecation log captured for the representative request set (`/api/callback`, `/api/healthcheck`, `/api/twitter/highlights`) and archived in the PR description.
- 12/12 phpunit pass count recorded in the Stage 0 commit message as the baseline.

### Stage 1 specific

- Each `SymfonySetList::SYMFONY_<n>` Rector set committed separately; PR description links to each diff.
- **Local `bin/console` smoke + production canary deploy** — there is no staging environment. The smoke set covers (at minimum) the three `api.yaml` endpoints; production canary is a partial deploy with metric watch before full rollout.

### Stage 2 specific

- **Mapping consistency:** `bin/console doctrine:schema:validate --skip-sync` clean — validates the ORM mapping is internally consistent regardless of DB state. Safe to run in test env (empty SQLite).
- **No-drift check (run against a production-shaped DB, e.g. during canary):** `bin/console doctrine:schema:update --dump-sql --no-interaction` returns **no SQL** — meaning the ORM 3 attribute-mapped entities derive the same schema the live DB already has. A non-empty dump means the conversion shifted an inferred mapping; stop and investigate.
- `DoctrineSetList::ANNOTATIONS_TO_ATTRIBUTES` Rector run committed separately; diff reviewed entity-by-entity.
- One representative end-to-end repository check per converted entity: load → modify → persist → flush → reload. Can be ad-hoc via `bin/console` or as a small temporary test.

### Stage 3 specific

- Provisioning runtime confirmed at PHP ≥ 8.2 **before** the Symfony version bump within the stage.
- `composer.json` `config.platform.php` matches the deployed runtime version exactly.
- PHPStan analysis runs on PHP 8.2+ ahead of merge.

## 7. Branch strategy and rollback

### Branches

- One branch per stage, off `main`. Suggested names: `migration/s0-preflight`, `migration/s1-symfony-6.4`, `migration/s2-doctrine-3`, `migration/s3-symfony-7.4`.
- Within each stage, commits stay small: Rector application + diff, manual touch-ups, composer.lock bump, config changes — separate commits to keep mid-stage `git bisect` viable.
- Existing `http-api` working branch is unrelated and continues independently; rebase it after each stage merges.

### Rollback model

- **Code & dependencies.** `git revert` of the stage's merge commit. `composer.lock` is one file and atomic; reverting it restores the prior dependency tree, and a `composer install` pulls `vendor/` back in line.
- **Database schema.** Stage 2 is the only stage that touches Doctrine mapping. The exit gate requires `doctrine:schema:validate --skip-sync` clean — meaning **no schema change** results from the attribute conversion. Stage 2 revert is therefore code-only. Stop the stage if validate reports drift.
- **Runtime.** Stage 3's PHP 8.2 floor is the only runtime change. The provisioning bump lands as its own commit ahead of the version bump within the stage; reverting the framework bump leaves the runtime bump in place, so rollback is single-file revert of the framework commit (PHP 8.1 → 8.2 is backward-compatible with Symfony 6.4).

### Per-stage risk profile

| Stage | Headline risk | Mitigation |
|---|---|---|
| 0 | Very low — cleanup only | PHPStan/phpunit baselines locked at exit |
| 1 | **Medium-high — auth flows.** `security.yaml` authenticator-system port; auth has thin test coverage. | Dedicated local `bin/console` smoke on auth-touching endpoints; production canary; auth-flow regression block the merge until manually exercised |
| 2 | **Medium-high — mapping/DB.** Annotation→attribute conversion can shift inferred mapping. | `doctrine:schema:validate --skip-sync` clean (mapping consistency) **and** `doctrine:schema:update --dump-sql` returns empty against a prod-shaped DB (no drift); revert if either fails |
| 3 | Low-medium — runtime + framework | Provisioning bump as its own commit ahead of version bump; PHPStan run on PHP 8.2+ ahead of merge |

### No feature flags

Migration changes touch the framework underneath every feature. Flags don't give a useful seam. Per-stage merge + production canary is the unit of safety.

## 9. Open coordination items

- **Provisioning ownership.** The same person handles app + provisioning end-to-end (confirmed during brainstorming). Stage 3 is therefore a single PR rather than a coordination handoff.
- **Production canary mechanics.** This design assumes a production canary capability exists (partial-fleet deploy + metric watch + rollback). If it doesn't, Stage 1 and Stage 3 gates need to be revised — a follow-up before Stage 1 starts.
- **Test coverage compensation.** Auth flows are the thinnest area. Stage 1's local `bin/console` smoke is the explicit compensator; consider adding one functional test that exercises authenticated and unauthenticated flows during Stage 1 itself, as a forward-looking deposit on the test-coverage debt.
